# Merkle-Poseidon-ECDSA signature Verification in Noir
This circuit creates a verifiable proof, that a ECDSA (secp256k1 curve) signature over a Poseidon-Merkle-root from an encoded RDF dataset is verified correctly.
Idea: Use this proof as a trust anchor for subsequent selective disclosure of RDF and SPARQL query proofs over a signed dataset.

Pre-processing of RDF dataset happens in Rust 
- canonicalization
- transform quads into Field elements with SHA256
- construct dataset merkle root with Poseidon and sign root with ECDSA
Warning: Make sure to sign the raw root (do not hash the root before signing in Rust) sothat it matches Noir's verify_signature blackbox function.
- verification check and writing the Prover.toml file required for Noir circuit


# Flow
Issuer -> Holder/Prover -> Verifier

Issuer: Issues 
- pre-processed RDF dataset
- generates keypair for ecdsa curve
- ecdsa signature over computed dataset poseidon-merkle root

Issuer passes to Prover: 
- issuer Public Key (verifier can look up who the signature came from)
- Signature
- Merkle root
- Dataset "leaves" (pre-processed RDF encoded as Field elements)

Prover: 
- generates proof in-circuit: “There exists a signature σ and a canonicalized RDF dataset, D s. t.
- assertion 1: re-computes Merkle root (R') = Merkle root (R)
- assertion 2: root was signed by the issuers ECDSA key s. t. Verify(pk, m, σ) = true

Prover passes to Verifier:
- proof
- verification key
- public inputs

Verifier: Verifies proof with Barettenberg backend (Ultra-Honk)

### Circuit details

Public inputs:

- `pub_key_x: [u8; 32]`
- `pub_key_y: [u8; 32]`
- `disclosed_h_terms: [Field; 2]`

Private inputs:

- `root: Field`
- `signature: [u8; 64]`
- `leaves: [Field; 4]`
- `all_h_term_sorted_like_rdf_lex: [[Field; 5]; 4]`

circuit computes:   
- Merkle tree with fixed depth 2 from dataset leaves with Poseidon(5) hash over BN254 = R'
- assert  R' = R  
- assert  ECDSA verification (pk, R', signature) = true using Noir's blackbox function verify_signature
- assert  each public disclosed term hash appears at least once somewhere in some position of some quad

circuit output:
- proof
- verification key
- public inputs: disclosed, hashed terns, Issuer public key

# Limitations
- Tree size is fixed
- Ordered tree construction (RDF datasets are unordered)
- No selective disclosure yet

What it does not prove yet:

- that disclosed terms came from a specific quad position
- that both disclosed terms come from the same quad
- that the verifier is checking against a public root
- any general-size Merkle tree
- selective disclosure of unhashed RDF values

## Expected preprocessing

This circuit assumes preprocessing happened outside Noir. The witness in `Prover.toml` is expected to be prepared as follows:

1. Canonicalize the RDF dataset externally.
2. Convert RDF terms into `Field` elements externally.
3. Sort the quads in the same order used by `all_h_term_sorted_like_rdf_lex`.
4. Hash each quad with Poseidon-5 to get 4 leaf values.
5. Build the 4-leaf Poseidon Merkle root externally.
6. Sign the raw 32-byte big-endian root bytes that Noir verifies via `std::ecdsa_secp256k1::verify_signature`.

The circuit does not perform RDF canonicalization or RDF term hashing on its own.
